\chapter{Metodyka badań}
Piąty rozdział prezentuje metodykę badań przeprowadzonych w ramach pracy, w~tym cel i zakres badań oraz szczegółowy opis dwóch scenariuszy testowych: scenariusza $\mathcal{A}$, opartego na integracji z bazą danych ze sterownikiem blokującym i reaktywnym oraz scenariusza $\mathcal{B}$, opartego na agregacji wyników opartej na mechanizmie kworum, prezentując dla każdego z nich przyjęte założenia i opisując ich implementację w ramach wybranych modeli współbieżności. Rozdział definiuje również hipotezy badawcze, określone na podstawie przyjętych pytań i celów badawczych, zdefiniowanych scenariuszy testowych oraz~przeanalizowanej literatury.

\section{Cel i zakres badań}

Celem niniejszej pracy jest analiza porównawcza oraz zbadanie możliwości integracyjnych mechanizmów wprowadzonych w ramach Project Loom, w szczególności wirtualnych wątków oraz mechanizmu współbieżności strukturalnej, z istniejącymi modelami współbieżności w środowisku JVM. Badania koncentrują się na weryfikacji wydajności oraz właściwości implementacyjnych tych rozwiązań w kontekście dwóch scenariuszy badawczych:
\begin{itemize}
    \itemsep0em
    \item Scenariusz $\mathcal{A}$ (integracja z bazą danych ze sterownikiem blokującym i reaktywnym), który dotyczy analizy operacji blokujących wejścia i wyjścia przy wykorzystaniu różnych sterowników umożliwiających nawiązania połączenia z bazą danych: standardowego sterownika blokującego (JDBC) oraz jego reaktywnego odpowiednika (R2DBC),
    
    \item Scenariusz $\mathcal{B}$ (agregacja wyników oparta na mechanizmie kworum), który obejmuje implementację procesu równoległego pobierania danych z wielu źródeł, gdzie wynik końcowy jest determinowany przez odpowiedzi od określonej liczby serwisów; scenariusz ten służy do oceny mechanizmów koordynacji zadań i~zarządzania ich cyklem życia.
\end{itemize}
Weryfikacja wydajnościowa została przeprowadzona z wykorzystaniem narzędzia JMH (Java Microbenchmark Harness) \cite{jmh-documentation}. Szczegółowy opis użytych trybów benchmarków i ich konfiguracji pod kątem przeprowadzonych badań został uwzględniony w rozdziale szóstym.

\subsection{Szczegółowe cele badawcze}
W ramach pracy zdefiniowano następujące cele szczegółowe:
\begin{enumerate}
    \itemsep0em

    \item \textbf{ocena charakterystyki wydajnościowej}: porównanie przepustowości oraz opóźnień wirtualnych wątków z istniejącymi modelami współbieżności w~oparciu o wyniki uzyskane z~benchmarków JMH,
    
    \item \textbf{ocena wpływu mechanizmów Project Loom na model programowania współbieżnego}: analiza zmian w sposobie strukturyzacji przepływu sterowania przy przejściu z~paradygmatu asynchronicznego i reaktywnego na imperatywny model oparty na wirtualnych wątkach,

    \item \textbf{badanie kompatybilności z istniejącymi bibliotekami i frameworkami}: weryfikacja łatwości integracji mechanizmów Project Loom z obecnym ekosystemem oraz identyfikacja potencjalnych ograniczeń w~ich współużytkowaniu,

    \item \textbf{analiza jakościowa kodu pod kątem złożoności implementacyjnej}: porównanie struktury kodu powstałego w ramach implementacji scenariuszy $\mathcal{A}$ i $\mathcal{B}$ ze szczególnym uwzględnieniem czytelności zapisu logiki biznesowej oraz trudności w utrzymaniu kodu w różnych modelach współbieżności,

    \item \textbf{weryfikacja wbudowanych mechanizmów odporności na błędy}: ocena mechanizmów współbieżności strukturalnej w zakresie automatyzacji obsługi błędów, propagacji przerwań oraz eliminacji ryzyka wycieków danych w~porównaniu do standardowych implementacji opartych o istniejące modele współbieżnego przetwarzania danych,

    \item \textbf{sformułowanie rekomendacji technicznych}: opracowanie wytycznych dotyczących doboru odpowiedniego modelu współbieżności w zależności od specyfiki problemu obliczeniowego oraz wymagań systemowych dotyczących skalowalności i czasu odpowiedzi.
\end{enumerate}

\section{Szczegółowy opis scenariuszy testowych}

\subsection{Scenariusz \ensuremath{\mathcal{A}}: Integracja z~bazą danych ze~sterownikiem blokującym (JDBC) i~reaktywnym (R2DBC)}
\label{sec:description-scenario-A}

Scenariusz $\mathcal{A}$ ma na celu zbadanie wpływu doboru modelu współbieżności na integrację z bazą danych przy wykorzystaniu różnych sterowników odpowiedzialnych za komunikację z bazą danych, ze szczególnym uwzględnieniem zastosowania mechanizmu wirtualnych wątków, oferowanego w ramach Project Loom.

W środowisku JVM wyróżnia się dwa główne sterowniki opdowiedzialne za nawiązanie połączenia z bazą danych:
\begin{itemize}
    \itemsep0em
    \item JDBC (ang. \emph{Java Database Connectivity}), oferujący standardowe API, w~którym operacje są wykonywane w sposób blokujący \cite{jdbc-documentation},
    \item R2DBC (ang. \emph{Reactive Relational Database Connectivity}), oferujący API zaprojektowane do podejścia reaktywnego, w którym operacje na bazie są nieblokujące i oparte na strumieniach danych \cite{r2dbc-documentation}.
\end{itemize}
W ramach eksperymentu skonstruowano trzy warianty reprezentujące różne modele współbieżności:
\begin{itemize}
    \itemsep0em
    \item \texttt{JDBC+PT}, wykorzystujący sterownik JDBC przy wykorzystaniu wątków platformowych,
    \item \texttt{R2DBC}, wykorzystujący sterownik R2DBC oraz reaktywny model współbieżności,
    \item \texttt{JDBC+VT}, wykorzystujący sterownik JDBC, lecz przy wykorzystaniu wątków wirtualnych, oferowanych przez Project Loom.
\end{itemize}
Badanie przeprowadzono dla każdego z powyższych wariantów. Wyniki eksperymentu przeprowadzonego w ramach scenariusza $\mathcal{A}$ przedstawiono w rozdziale szóstym, w sekcji \hyperref[sec:results-scenario-A]{6.3.1 Wyniki pomiarów w ramach scenariusza \ensuremath{\mathcal{A}}}.

\begin{itemize}
    \itemsep0em

    \item dla wariantu \texttt{R2DBC}: liczbę równocześnie aktywnych subskrypcji,

\end{itemize}
Z~racji, że scenariusz jest badany pod kątem różnych poziomów współbieżności, w~każdym wariancie wszystkie wątki rozpoczynają wykonywanie zapytania w tym samym momencie, aby nie zaburzyć wyniku z powodu powolnego rozłożenia startu, a także zadbano o poprawne zamknięcie połączenia z bazą po wykonaniu każdego zapytania.

    var connectionFactory = getConnectionFactoryForR2DBC();

    var blackhole = getBenchmarkBlackhole();

    void executeQueryUsingR2DBC(String query) {
        return Mono.usingWhen(
                connectionFactory.create(),
                connection -> {
                    Flux.from(connection.createStatement(query)
                        .bind("<ID>", id)
                        .execute()
                    )
                    .flatMap(result -> 
                        result.map((row, _) -> {
                            var value =
                                row.get(COLUMN_NAME, String.class);

                            ...
                            return new Object();
                        })
                    )
                    .then()
                },
                Connection::close
        );
    }

\subsection{Scenariusz \ensuremath{\mathcal{B}}: Agregacja wyników oparta na mechanizmie kworum}
\label{sec:description-scenario-B}

Scenariusz $\mathcal{B}$ ma na celu zbadanie wpływu doboru mechanizmu oferującego współbieżne przetwarzanie danych, ze szczególnym uwzględnieniem mechanizmu współbieżności strukturalnej, w kontekście zadania polegającego na agregacji wyników opartej na mechanizmie kworum (ang. \emph{quorum mechanism}). Mechanizm ten polega na podejmowaniu decyzji lub uznaniu operacji za zakończoną dopiero po uzyskaniu określonej liczby odpowiedzi (tzw. \emph{wartość kworum}), zamiast oczekiwania na wszystkie odpowiedzi.

W analizowanym scenariuszu rozpatrywany jest zbiór pięciu niezależnych operacji, reprezentujących wywołania zewnętrznych serwisów. Operacje te są inicjowane równolegle i wykonywane współbieżnie. Każda z nich może zakończyć się sukcesem, zwracając wynik, lub niepowodzeniem. Celem przetwarzania jest uzyskanie co~najmniej trzech poprawnych wyników (tj. wartość kworum jest równa $3$). Osiągnięcie tego progu traktowane jest jako warunek sukcesu całego procesu agregacji. W~momencie spełnienia tego warunku dalsze oczekiwanie na pozostałe wyniki zostaje przerwane, a~wszystkie niezakończone operacje są anulowane. W~przypadku, gdy liczba poprawnie zakończonych operacji jest mniejsza niż wartość kworum, proces uznawany jest za zakończony niepowodzeniem.

\subsubsection{Opis symulowanych serwisów}
Warianty scenariusza testowego $\mathcal{B}$ wykorzystują różne typy serwisów. Każdy serwis jest definiowany przez zestaw następujących parametrów:
\begin{itemize}
    % \itemsep0em
    \item $D_{\text{base}}$ -- wartość bazowego opóźnienia odpowiedzi serwisu, podana w milisekundach i wyznaczana z zadanego przedziału (ang. \emph{base delay}),
    \item $J_{\text{base}}$ -- wartość odchylenia od bazowego opóźnienia, podana w milisekundach i wyznaczana z zadanego przedziału (ang. \emph{jitter}),
    \item $P_{\text{outlier}}$ -- prawdopodobieństwo wystąpienia bardzo wolnej odpowiedzi, gdzie $0 \leqslant P_{\text{outlier}} \leqslant 1$ (ang. \emph{outlier probability}),
    \item $L_{\text{outlier}}$ -- wartość opóźnienia podczas wystąpienia bardzo wolnej odpowiedzi, podana w milisekundach (ang. \emph{outlier latency}).
\end{itemize}
Na potrzeby eksperymentu wartości bazowego opóźnienia ($D_{\text{base}}$) i odchylenia od tej wartości ($J_{\text{base}}$) oraz ustalenie wystąpienia bardzo wolnej odpowiedzi ($P_{\text{outlier}}$) są wyznaczane losowo przy użyciu ustalonej wartości ziarna, co gwarantuje deterministyczne odtworzenie dokładnych wartości opóźnień symulowanych serwisów dla każdego wariantu badanego modelu współbieżności.

Tabela \ref{table:scenario-B-services-types} przedstawia cztery typy serwisów wraz z ich parametrami, symulujące różne rodzaje opóźnień i czasu przetwarzania. Serwisy \texttt{FAST} i \texttt{MEDIUM} charakteryzują się odpowiednio krótkim i umiarkowanie krótkim czasem odpowiedzi przy małym prawdopodobieństwie wystąpienia bardzo wolnej odpowiedzi. Serwis \texttt{SLOW} cechuje długi czas odpowiedzi przy nieco większym prawdopodobieństwie wystąpienia bardzo wolnej odpowiedzi niż poprzednie serwisy. Odmiennym zachowaniem charakteryzuje się serwis \texttt{OUTLIER\_PRONE}, który bazowo zwraca odpowiedź w umiarkowanie krótkim czasie, lecz posiada największe prawdopodobieństwo wystąpienia bardzo wolnej odpowiedzi w porównaniu do pozostałych serwisów.

\subsubsection{Warianty modeli współbieżności i ich implementacja}

W ramach eksperymentu zdecydowano się na zbadanie trzech wariantów umożliwiających implementację logiki zawartej w scenariuszu $\mathcal{B}$:
\begin{itemize}
    % \itemsep0em
    \item \texttt{Legacy}, wykorzystujący w tym celu interfejs \mintinline{java}|ExecutorService| i natywne narzędzia upraszczające współbieżne przetwarzanie danych, w tym klasę \mintinline{java}|CompletableFuture|,
    \item \texttt{Reactive}, wykorzystujący w tym mechanizmy programowania asynchronicznego, dostępne w ramach narzędzia Reactive Streams, 
    \item \texttt{Loom}, wykorzystujący w tym celu mechanizm współbieżności strukturalnej w~oparciu o własną strategię synchronizacji podzadań.
\end{itemize}
Badanie przeprowadzono dla każdego z powyższych wariantów na każdym z czterech zestawów serwisów, opisanych w poprzednim akapicie. Wyniki eksperymentu przeprowadzonego w ramach scenariusza $\mathcal{B}$ przedstawiono w rozdziale szóstym, w~podrozdziale \hyperref[sec:results-scenario-B]{6.3.2 Wyniki pomiarów w ramach scenariusza \ensuremath{\mathcal{B}}}.

    String aggregateQuorum(
        List<CompletableFuture<String>> futures,
        int quorum
    ) {
        var successes = new ArrayList<String>(quorum);
        for (var future : futures) {
            try {
                var result = future.get();
                successes.add(result);
                if (successes.size() >= quorum) {
                    return aggregate(successes);
                }
            } catch (Exception e) { ... }
        }

        if (successes.size() >= quorum) {
            return aggregate(successes);
        }
        throw new IllegalStateException("Quorum not achieved");
    }

W przeciwnym wypadku zebrane wyniki są agregowane do pojedynczego wyniku tekstowego i~zwracane synchronicznie za pomocą metody \mintinline{java}|block|.

    var quorum = 3;
    var services = getServices(EXAMPLE_PRESET);

    return Flux.fromIterable(services)
                .flatMap(service -> 
                    Mono.fromCallable(service::call)
                        .subscribeOn(Schedulers.boundedElastic())
                        .onErrorResume(_ -> Mono.empty()),
                    services.size()
                )
                .take(quorum)
                .collectList()
                .flatMap(results -> {
                    if (results.size() < quorum) {
                        return Mono.error(
                            new IllegalStateException(
                                "Quorum not achieved"
                            )
                        );
                    }
                    return Mono.just(aggregate(results));
                })
                .block();
    \end{minted}

Po zebraniu wyników za pomocą metody \mintinline{java}|join|, w przypadku uzyskania wymaganej wartości kworum następuje agregacja i zwrócenie wyniku, a~w~przeciwnym razie następuje zwrócenie błędu agregacji.
Użycie konstrukcji \texttt{try}-with-resources zapewnia automatyczne przerwanie wywołań serwisów po uzyskaniu wyniku (patrz linia 5. w listingu \ref{listing:scenario-B-loom-code-1}).

    public final class QuorumJoiner
    implements StructuredTaskScope.Joiner<String, List<String>> {
        
        private final int quorum;
        private final ConcurrentLinkedQueue<String> successes;
        private final AtomicInteger successCount;

        public QuorumJoiner(int quorum) { ... }

        @Override
        public boolean onComplete(
            Subtask<? extends String> subtask
        ) {
            if (subtask.state() == Subtask.State.SUCCESS) {
                successes.add(subtask.get());
            }
            return successCount.incrementAndGet() >= quorum;
        }

        @Override
        public List<String> result() {
            return List.copyOf(successes);
        }
    }
    \end{minted}
